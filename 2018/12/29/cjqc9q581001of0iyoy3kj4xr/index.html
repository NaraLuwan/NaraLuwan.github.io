字段都比较简单，但是为什么没有主键ID？猜想应该是在基类 BaseEntity 里，来看下：

BaseEntity

  • 果然主键在基类里定义的。@GeneratedValue(strategy = GenerationType.IDENTITY) 由数据库生成，自增且全局唯一。在 Apollo 整个项目中所有的实体类都继承自 BaseEntity。
  • @MappedSuperclass 注解基于代码复用和模型分离的思想，可以将实体类的多个属性分别封装到不同的非实体类中。这里的 BaseEntity
    类将不是一个完整的实体类，不会映射到数据库表，但是它的属性都将映射到其子类的数据库表字段中。
  • @Inheritance(strategy = InheritanceType.TABLE_PER_CLASS) 这种策略支持双向的一对多关联，每个子类都对应一张数据库表。
  • @PrePersist、@PreUpdate、@PreRemove 注解，CRD 操作前，设置对应的时间字段。

为什么需要同步

在文初的流程图中，可以看到 App 创建时，在 Portal Service 存储完成后，会异步同步到 Admin Service 中，为什么不用同一张表呢？

在 Apollo 的架构中，一个环境( Env ) 对应一套 Admin Service 和 Config Service 。
而 Portal Service 会管理所有环境( Env ) 。因此，每次创建 App 后，需要进行同步。

或者说，App 在 Portal Service 中，表示需要管理的 App 。而在 Admin Service 和 Config Service 中，表示存在的 App 。

思考

  • Portal 在同步 App 到 Admin Service 时，如果发生网络异常，同步失败。那么此时会出现该 App 存在于 Portal ，却不存在于 Admin Service 中。
  • 新增了一套环境( Env ) ，也会导致 Portal 和 Admin Service 不一致的情况。
